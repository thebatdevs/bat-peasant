# `bat-peasant` coding agent

I want to build `theBatPeasant`, a personalized coding agent system for my TypeScript AWS Serverless projects.

My goal is not to build a new LLM or coding agent from scratch. My goal is to create a repo-native AI control layer that makes existing coding agents like Codex, GitHub Copilot Agent Mode, Claude Code, Cursor, or Windsurf behave like a trained engineer who understands my coding style, architecture, project structure, domain logic, testing standards, and verification workflow.

## The goal of this project

I want to create a `./ai` directory in my repositories that stores all important context an AI coding agent needs to work reliably.

The `./ai` directory should help the agent understand:

- project architecture
- coding standards
- testing standards
- business/domain logic
- DynamoDB access patterns
- Serverless function patterns
- API handler/service/repository flow
- Zod schema patterns
- error handling and logging style
- examples of good code from my project
- required verification commands
- feature implementation workflows
- bug-fix workflows
- code review workflows

## Desired behavior

When user give a prompt like

> Add a new endpoint to create a user.

or:

> Fix the bug in the user creation flow.

The agent should not immediately start coding.

It should follow this workflow:

1. Understand the requirement.
2. Read `AGENTS.md`, `.github/copilot-instructions.md`, and the `./ai` directory.
3. Analyze the existing codebase and relevant domain docs.
4. Find existing similar patterns before creating new code.
5. Explain how the requested change fits into the current architecture.
6. Create a functional implementation plan and ask for confirmation.
7. Create a coding-level plan listing files, functions, schemas, tests, and commands, then ask for confirmation.
8. Implement the feature or bug fix according to my standards.
9. Create a test plan and ask for confirmation.
10. Implement tests using my Vitest conventions.
11. Run targeted verification first.
12. Run full verification before finalizing.
13. If tests fail, analyze the failure, fix the issue, and rerun verification.
14. Final response should summarize changed files, tests added, commands run, passing/failing status, and remaining risks.
